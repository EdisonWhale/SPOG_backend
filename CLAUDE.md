# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

FastAPI backend for **SPoG** (Single Pane of Glass), Highmark's centralized registry/governance platform for AI agents. It lets teams register "Projects" (business initiatives) and the "Agents" that belong to them, tracks ownership/approval status, and syncs promotion state with GitLab. Deployed on GCP Cloud Run; data lives in Firestore.

`AGENTS.md` and `docs/PROJECT_INTAKE_IMPLEMENTATION.md` describe an earlier/aspirational shape of this project (`src/routes`, `src/common`, `src/tests`, strict TDD) that **does not match the current source tree**. Treat the code under `src/app/` as ground truth over those two docs.

## Commands

There is no Makefile, Dockerfile, CI config, or lint/test tooling wired up yet. Dependencies are plain `pip`:

```bash
pip install -r src/requirements.txt
```

Run the dev server (must run with `src/` as the working directory / on `PYTHONPATH`, since all imports are `app.*` relative to `src/`):

```bash
cd src && python main.py          # uvicorn on 0.0.0.0:8081, reload off
# or
cd src && uvicorn main:app --reload --port 8081
```

Config is environment-file driven (`pydantic-settings`, see Config below). Locally this resolves to `src/app/config/.env.local`, which is already committed with non-secret defaults — no extra setup needed for a first run. Auth is disabled by default for local dev (`OAUTH__enabled`/`OAUTH__verify_signature` in `.env.local`), so unauthenticated requests get treated as `system@email.com` (see Auth below).

There is no pytest suite (`src/requirements.txt` lists `pytest` but there is no `tests/` directory). The closest thing to tests are ad-hoc, throwaway repro scripts at `src/_check.py` / `src/_repro.py` — run directly with `python src/_check.py` from `src/` to manually exercise a model/validator. When adding real tests, `fastapi.testclient.TestClient` against `main.app` is the natural entry point.

No `ruff`/`flake8`/`mypy` config exists in the repo; `uvx ruff check .` from `src/` works ad hoc if you want a quick lint pass.

OpenAPI spec is served at `GET /openapi.yaml` (YAML) or the default FastAPI `/openapi.json`; both are generated by `custom_openapi()` in `main.py`, which post-processes the schema through `app/utils/openapi_compliance.py` to satisfy an internal API-governance linter (security scheme shape, tag/description requirements, etc.) — don't hand-edit the raw schema, fix it via that normalizer.

## Architecture

**Layering:** `routers/` (HTTP, FastAPI `APIRouter`s) → `services/` (business logic, orchestration) → `repositories/` (Firestore data access, one per entity) → `AsyncDocumentRepository` (generic async Firestore CRUD/pagination base every repository extends) → `models/` (Pydantic).

**Entry point (`src/main.py`):** builds the `FastAPI` app from `api_metadata.py`, registers all routers with their path prefixes, sets up structured logging (Cloud Logging when `K_SERVICE`/`JOB_ENV` is set, otherwise a stream handler), and initializes `FastAPICache` (in-memory) in the `lifespan`. CORS middleware is only added when not running on Cloud Run (`K_SERVICE` unset). Router registration order matters: the draft router is included before the non-draft one so `/projects/draft/...` isn't swallowed by `/projects/{project_id}`.

**Config (`app/config/config.py`):** a `pydantic-settings` `BaseSettings` tree (`Config` → nested `GoogleCloudPlatformConfig`/`FirestoreConfig`/`GoogleCloudStorageConfig`/`GenAIConfig`/`JWTConfig`/`EntraConfig`/`CacheConfig`/`GitLabConfig`, double-underscore env nesting, e.g. `GCP__firestore_config__database_id`). `load_config()` picks the env file by `K_SERVICE` (Cloud Run revision name) or `JOB_ENV`, falling back to `local`, and loads `app/config/.env.<that>`. Several fields ship with hardcoded non-empty defaults (`jwt_config.aitr_jwt_signing_secret`, `cursor_secret_key`) meant purely as local-dev fallbacks — real deployments must override them via env, don't treat the in-code defaults as production-safe.

**Firestore document models:** every persisted entity derives from `SpogBaseModel` (`app/models/base/SpogBaseModel.py`), which composes `CustomBaseModel` (camelCase wire aliasing via `alias_generator=to_camel`, `to_dict()`/`to_response_dict()` helpers) with mixins from `app/models/base/CustomBaseModel.py`: `TimestampedModel` (created/updated), `TaggableModel`, `AuthorizableModel` (owning `author` email + validator), `DeprecatableModel`. Each concrete model (`Project`, `Agent`, `UseCase`, ...) sets its own `COLLECTION_NAME`.

**Create/Update model variants:** `Project`/`DraftProject` derive their `*Create`/`*Update` request models via the `partial_model()` factory (`app/models/utils/partial_model.py`), which rebuilds a model with selected fields made optional and/or excluded while preserving field constraints and validators — this is the established pattern for new partial-update models. `UseCaseUpdate.py` is a hand-rolled exception to this pattern (not generated via `partial_model()`) and has drifted out of sync with `UseCase.py` as a result — prefer `partial_model()` over hand-rolling when adding new Update models.

**Draft vs. published entities:** `Project`/`Agent` (collections `Projects`/`Agents`) have parallel `DraftProject`/`DraftAgent` counterparts (collections `DraftProjects`/`DraftAgents`). Drafts and their published counterparts share the same IDs by design so a draft can be "published" (promoted) later; `ProjectIntakeService` owns both flows behind `project_intake_router.py` (published) and `draft_project_router.py` (drafts) — the two routers are intentionally near-mirrors of each other (same error-response conventions), so when changing one, check whether the other needs the equivalent change.

**Pagination:** cursor-based, implemented once in `AsyncDocumentRepository.get_all_paginated()` and reused by every repository. The cursor is an encrypted/opaque blob (`CursorData` model + `app/utils/cursor_utils.py`'s Fernet-based `cursor_encoder`, keyed by `Config.cursor_secret_key`) carrying page/direction/boundary-doc-id — don't construct or parse cursors by hand.

**Auth — two independent mechanisms exist; only one is live:**
- Live: `app/core/auth.py` validates Microsoft Entra (Azure AD) bearer tokens against the tenant JWKS and exposes `get_current_author`/`get_current_user_info` as FastAPI dependencies; this is what every router actually uses. When `EntraConfig`/`OAuthConfig.enabled` is false (local dev), requests without a token are treated as `system@email.com` instead of being rejected.
- Dead: `app/services/AuthService.py` (HS256 JWT via an `aitr_jwt` cookie, `Config.jwt_config`) is not imported by any router — it's a leftover from an earlier auth design. Don't extend it by habit; new auth work belongs in `app/core/auth.py`.

**Authorization / ownership:** `ProjectIntakeService._assert_owner()` raises `NotAuthorizedError` (defined locally in `ProjectIntakeService.py`, not in the shared `app/core/exceptions.py` hierarchy) when the acting author doesn't match an entity's stored `author`. Routers that mutate owner-scoped resources must catch `NotAuthorizedError` → 403 in addition to the shared `ServiceException` subclasses in `app/core/exceptions.py` (`ValidationException`/`NotFoundException`/`ConflictException`/`SystemException` → 400/404/409/500) — when adding a new mutating endpoint, copy the except-block pattern from a sibling owner-scoped endpoint in the same router, not from a read-only one.

**GitLab integration:** `app/services/GitlabService.py` talks to the GitLab API (via `Config.gitlab_config`). Two separate flows use it:
- `project_intake_router.py`'s `POST /{project_id}/sync-project-status` — on-demand, owner-triggered, tracks state via `Project.gitlab_sync_requested_status` / `gitlab_sync_requested_at`.
- `gitlab_sync_router.py`'s `POST /internal/gitlab/sync-staging-to-production` — a Cloud Scheduler-triggered batch job (bearer-token-gated by `SCHEDULER_SECRET`) meant to promote all staging agents of approved projects to production. **This router reads/writes field names that don't exist on the current models** (`project.status`, `agent.code_version`, `agent.release_date` — the real fields are `Project.active`/`.locked` and `Agent.production_code_version`/`.latest_version_release_date`), and filters use cases by `get_by_status("approved")` (lowercase) against an enum whose real value is `"Approved"`. Every one of these mismatches throws, gets swallowed by a broad `except Exception`, and is logged rather than surfaced — so the endpoint currently always responds `no_approved_projects`/`no_staging_agents` and never promotes anything. If you touch this router, treat it as broken and reconcile it against `Project`/`Agent`'s actual fields before relying on it.

**Response schemas:** `app/models/schemas/CommonSchemas.py` centralizes shared response envelopes (`CommonErrorResponse`, `SuccessResponse`, `DeleteResponse`) and pre-built `responses=` dicts (`COMMON_ERROR_RESPONSES`, `COMMON_OWNED_ERROR_RESPONSES`) that routers spread into their `@router.<verb>(..., responses={**COMMON_..._RESPONSES, ...})` decorators for consistent OpenAPI error documentation — use `COMMON_OWNED_ERROR_RESPONSES` (includes 403) for any endpoint that can raise `NotAuthorizedError`.

## Conventions

- Filenames for models/repositories/services are PascalCase matching the single class they define (`ProjectRepository.py` → `class ProjectRepository`); routers/utils/enums/config modules are snake_case.
- Pydantic fields are snake_case in Python with camelCase on the wire (`CustomBaseModel`'s `alias_generator=to_camel`); build response bodies with a model's `.to_response_dict()` (or `.to_dict()` with explicit args), not raw `.model_dump()`, so alias/exclusion behavior stays consistent.
- Firestore-facing async methods run blocking SDK calls off the event loop via `ThreadPoolExecutor`/`run_in_executor` (see `CloudStorageService`) or rely on the `google-cloud-firestore` async client directly (`AsyncDocumentRepository`) — don't call the sync Firestore/GCS SDK inline in an `async def` without one of these.
